### Цель работы
1.  Изучить архитектурный стиль REST и принципы построения веб-сервисов.
2.  Освоить работу с HTTP-методами (GET, POST, PUT, PATCH, DELETE) и кодами состояний.

5.  Изучить основы работы с ORM (Object-Relational Mapping) для абстракции работы с данными.
6.  Реализовать механизм мягкого удаления (Soft Delete) данных.

### Технические ограничения
-   СУБД: PostgreSQL (версия 16 или выше).
-   Архитектура: Модульная (разделение на контроллеры, сервисы, модели/сущности, DTO).

-   Удаление данных: Физическое удаление записей из БД запрещено. Требуется использование Soft Delete.
-   Пагинация: Обязательна для эндпоинта получения списка ресурсов.
-   Миграции: Инициализация и изменение схемы базы данных должны производиться исключительно через код миграций. Ручное создание таблиц через SQL-клиенты запрещено.

## Описание API

| Метод   | URI             | Описание                                  | Успешный статус |
|---------|-----------------|-------------------------------------------|-----------------|
| GET     | `/books`        | Список книг с пагинацией (фильтр по status) | 200 OK          |
| GET     | `/books/{id}`   | Получить книгу по ID                      | 200 OK          |
| POST    | `/books`        | Создать книгу                             | 201 Created     |
| PUT     | `/books/{id}`   | Полное обновление книги                   | 200 OK          |
| PATCH   | `/books/{id}`   | Частичное обновление                      | 200 OK          |
| DELETE  | `/books/{id}`   | Мягкое удаление                           | 204 No Content  |

### Параметры пагинации `GET /books`:
- `page` (int, default 1, min 1) — номер страницы
- `limit` (int, default 10, min 1, max 100) — записей на страницу
- `status` (str, optional) — фильтр: `draft` или `published`

Ответ списка содержит `data[]` и `meta` с полями `total`, `page`, `limit`, `total_pages`.

## Контрольные вопросы (для защиты лабораторной работы)

### 1. В чем заключается основная концепция REST?
REST (Representational State Transfer) — архитектурный стиль, в котором сервер предоставляет доступ к ресурсам через унифицированные URL, а клиент взаимодействует с ними, используя стандартные методы HTTP (GET, POST, PUT, PATCH, DELETE). Сервер не хранит состояние клиента (stateless), каждый запрос самодостаточен.

### 2. Какие HTTP-методы являются идемпотентными? Почему это важно для надежности API?
Идемпотентные методы: GET, PUT, DELETE, HEAD. Это значит, что повторный вызов с теми же параметрами не изменит результат (побочный эффект тот же). Это важно для надёжности: можно безопасно повторить запрос при сбоях сети.

### 3. В чем семантическая разница между методами PUT и PATCH?
- **PUT** — полная замена ресурса; клиент обязан прислать все поля.
- **PATCH** — частичное изменение; клиент передаёт только те поля, которые нужно обновить.

### 4. Какие коды состояния HTTP следует возвращать при успешном создании ресурса, успешном удалении и ошибке валидации данных?
- 201 Created — при успешном создании ресурса
- 204 No Content — при успешном удалении
- 400 Bad Request — при ошибках валидации данных

### 5. Что такое Soft Delete? Какие преимущества и недостатки оно имеет по сравнению с физическим удалением (Hard Delete)?
Мягкое удаление (Soft Delete) — запись не удаляется физически, а помечается (например, `is_deleted = true`).
- **Преимущества**: возможность восстановления, аудит, сохранение связанных данных.
- **Недостатки**: усложнение запросов (везде фильтровать `is_deleted = false`), увеличение объёма БД.

### 6. Зачем нужна пагинация данных в API? Какие существуют основные типы пагинации (offset-based, cursor-based)?
Пагинация нужна, чтобы не передавать тысячи записей за раз, снижая нагрузку на сервер и клиента.
- **Offset-based** (сдвиг и лимит) — простой, но может пропускать/дублировать записи при изменении данных
- **Cursor-based** (указатель на следующую страницу) — надёжнее, но сложнее в реализации

### 7. Зачем использовать DTO (Data Transfer Objects) вместо передачи сущностей базы данных напрямую в контроллер?
DTO позволяют отделить внутреннюю модель БД (с паролями, служебными полями) от того, что видит клиент. Это повышает безопасность и гибкость: можно менять структуру БД, не ломая API.

### 8. Какие преимущества дает использование ORM по сравнению с написанием "сырых" SQL-запросов?
- Абстракция от SQL: пишем на Python, а не на диалекте конкретной БД
- Безопасность: защита от SQL-инъекций
- Миграции схемы через код (Alembic)
- Кеширование сессий, автоматические связи
- Переносимость между разными СУБД

### 9. Как обеспечить безопасность чувствительных данных (пароли от БД) при контейнеризации приложения?
Пароли и ключи не зашиваются в код или Docker-образ, а передаются через переменные окружения (`.env` файл, не добавляемый в Git). В Docker Compose значения подставляются из файла или окружения хоста.

### 10. Что такое healthcheck в Docker Compose и зачем он нужен в связке приложений?
`healthcheck` проверяет готовность контейнера (например, `pg_isready` для PostgreSQL). Другие сервисы с `depends_on: condition: service_healthy` стартуют только после успешной проверки. Это гарантирует, что приложение не попытается подключиться к ещё не готовой БД.
